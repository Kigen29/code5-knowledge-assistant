# Client Project Operations Policy

## Purpose
Code 5 Developers delivers client work through disciplined planning, communication, and delivery control. This policy explains how project teams manage client expectations, approvals, meetings, reporting, and handover.

## Project Kickoff
Every client project must start with a kickoff covering objectives, scope, communication channels, stakeholders, milestones, risks, responsibilities, and approval paths. The project lead records kickoff decisions and shares them with the client.

## Communication Rhythm
Active projects must have a predictable communication rhythm. Standard cadence is a weekly status update, sprint or task review where applicable, and immediate escalation for blockers that threaten timeline, budget, security, or launch readiness.

## Status Reporting
Status reports should include completed work, upcoming work, risks, decisions needed, timeline movement, budget notes, and links to relevant demos or tickets. Reports must be plain enough for non-technical stakeholders to understand.

## Approvals
Client approvals must be captured in writing for scope changes, release dates, production deployments, major design decisions, third-party costs, and access to sensitive systems. Verbal approval should be followed by written confirmation.

## Handover
A maintenance handover must include repository links, deployment notes, environment variables, infrastructure ownership, admin accounts, monitoring links, backup procedures, support contacts, known issues, and open improvement items.

## Client Materials
Client documents, screenshots, code, credentials, and exports must be stored in approved project locations. Team members should avoid scattering client material across personal drives, chat threads, or untracked local folders.
